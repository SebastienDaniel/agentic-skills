You are an initialization tool that analyzes the current project and interactively builds a validation pipeline tailored to it. The pipeline consists of a shell script that runs outside the model context for efficiency, a thin skill wrapper, and project documentation stubs.

Follow the phases below exactly.

## Phase 1 — Project Analysis

Silently analyze the project. Do NOT output findings yet.

1. Search for project manifest and config files to detect languages, package managers, and tooling, for example:
   - `package.json`, `yarn.lock`, `package-lock.json`, `pnpm-lock.yaml`
   - `Cargo.toml`
   - `pyproject.toml`, `setup.py`, `setup.cfg`, `requirements.txt`, `Pipfile`
   - `go.mod`
   - `Gemfile`
   - `Makefile`, `CMakeLists.txt`
   - `composer.json`
   - `.csproj`, `*.sln`
   - `build.gradle`, `pom.xml`
   - `deno.json`, `bun.lockb`

2. When a manifest is found, read it to extract:
   - Available script/task commands (e.g. `scripts` in package.json, targets in Makefile, etc.)
   - Dev dependencies that indicate tooling (linters, formatters, type checkers, test runners, etc.)

3. Also scan for config files that reveal tooling, for example:
   - `.eslintrc*`, `eslint.config.*`, `.prettierrc*`, `prettier.config.*`
   - `tsconfig.json`, `jsconfig.json`
   - `.stylelintrc*`
   - `vitest.config.*`, `jest.config.*`, `cypress.config.*`, `playwright.config.*`
   - `.rustfmt.toml`, `clippy.toml`, `rustfmt.toml`
   - `.golangci.yml`, `.golangci.yaml`
   - `phpstan.neon`, `.php-cs-fixer.php`
   - `.rubocop.yml`
   - `mypy.ini`, `.flake8`, `ruff.toml`, `pyproject.toml [tool.*]`

4. From your findings, build a list of **suggested verification steps**. Each step has:
   - A **category name** (e.g. "Format", "Lint", "Type check", "Unit tests", "E2E tests", "Unused code", "Style lint", "Security audit", etc.)
   - A **suggested command** (e.g. `yarn run lint:check`)
   - A **rationale** (e.g. "Found eslint in devDependencies and a `lint:check` script")
   - Whether it is a **formatter** (auto-fixes files rather than reporting errors)

   Only suggest categories where you found concrete evidence of tooling. Do not guess. Ultra-think, take extra time to ensure the documented findings are used in the project.

## Phase 2 — Interactive Configuration

1. Present a brief summary of what you detected (languages, package manager, key tooling). Keep it to a few lines.

2. For each suggested verification step, use `AskUserQuestion` to ask the user **one at a time**:

   > **{Category name}**
   > I detected {rationale}.
   > Suggested command: `{command}`
   >
   > Options:
   > - Accept this command as-is
   > - Provide a different command
   > - Skip — does not apply

   Based on the user's response:
   - If accepted: add the step as-is to the final list.
   - If the user provides a different command: use their command instead.
   - If skipped: omit the step entirely.

3. After all suggested steps have been addressed, enter a loop:

   Use `AskUserQuestion` to ask:
   > **Additional commands**
   > Do you want to add any other verification commands?
   > If yes, provide a name and command (e.g. "Security audit: yarn audit").
   > Otherwise, reply "no" or "done".

   - If the user provides a command, add it to the final list and ask again.
   - If the user says no/done, exit the loop.

## Phase 3 — Output

### 3a. Generate the validation script

Create `.claude/scripts/validate.sh` — a self-contained shell script that runs all approved validation steps and produces structured output. The script runs outside the model context, saving tokens.

Use this template. Only include steps the user approved:

```bash
#!/usr/bin/env bash
# Auto-generated by /init-agentic-tools
# Project validation pipeline — runnable by any agent, CI, or developer terminal.

set -uo pipefail

# ── Step registry ──────────────────────────────────────────────
# Each step: name, command, and whether it's a formatter.
# Add or edit steps here — no other changes needed.

step_names=()
step_commands=()
step_formatter=()

{for each approved step, in order:}
step_names+=("{Category name}")
step_commands+=("{command}")
step_formatter+=({1 if formatter, 0 otherwise})

{end for}

# ── Step filtering ─────────────────────────────────────────────
# Pass step names as arguments to run only those steps (case-insensitive).
# No arguments = run all steps.

requested=("$@")

should_run() {
  local name_lower
  name_lower=$(echo "$1" | tr '[:upper:]' '[:lower:]')
  if [[ ${#requested[@]} -eq 0 ]]; then
    return 0
  fi
  for req in "${requested[@]}"; do
    local req_lower
    req_lower=$(echo "$req" | tr '[:upper:]' '[:lower:]')
    if [[ "$name_lower" == "$req_lower" ]]; then
      return 0
    fi
  done
  return 1
}

# ── Execution ──────────────────────────────────────────────────

any_failed=0
summary=""
total=0
passed=0
failed=0

for i in "${!step_names[@]}"; do
  name="${step_names[$i]}"
  command="${step_commands[$i]}"
  formatter="${step_formatter[$i]}"

  if ! should_run "$name"; then
    continue
  fi

  total=$((total + 1))
  output=""
  exit_code=0
  output=$(eval "$command" 2>&1) || exit_code=$?

  if [[ "$formatter" -eq 1 ]]; then
    # Formatters auto-fix files. Only fail if the command itself errors.
    if [[ $exit_code -eq 0 ]]; then
      summary+="## ${name} — PASS (formatter)"$'\n\n'
      passed=$((passed + 1))
    else
      any_failed=1
      failed=$((failed + 1))
      truncated=$(echo "$output" | tail -20)
      summary+="## ${name} — FAIL (formatter error)"$'\n'"${truncated}"$'\n\n'
    fi
  elif [[ $exit_code -eq 0 ]]; then
    summary+="## ${name} — PASS"$'\n\n'
    passed=$((passed + 1))
  else
    any_failed=1
    failed=$((failed + 1))
    # Keep output compact — last 30 lines covers most error summaries
    truncated=$(echo "$output" | tail -30)
    summary+="## ${name} — FAIL"$'\n'"${truncated}"$'\n\n'
  fi
done

# ── Output ─────────────────────────────────────────────────────

if [[ $any_failed -eq 0 ]]; then
  echo "Validation: ALL CLEAR (${passed}/${total} steps passed)"
else
  echo "Validation: ISSUES FOUND (${failed}/${total} steps failed)"
fi
echo ""
echo "$summary"

exit $any_failed
```

After writing the file, run `chmod +x .claude/scripts/validate.sh`.

### 3b. Generate the validate-changes skill

Create `.claude/skills/validate-changes/SKILL.md` — a thin wrapper that invokes the script:

```markdown
---
description: Run the project's validation pipeline and report results
argument-hint: step names to run (optional — omit to run all)
user-invocable: false
allowed-tools: Bash
---

Run the project's validation script and relay the output verbatim.

## Execution

Run: `"$CLAUDE_PROJECT_DIR"/.claude/scripts/validate.sh $ARGUMENTS`

Relay the script's stdout as your response. Do NOT fix errors — only report them.
The parent agent or user is responsible for fixes.
```

### 3c. Add validation guardrails to CLAUDE.md

Check the project's `.claude/CLAUDE.md`. If it does not already contain a validation guardrail section, append the following block:

```markdown

# Validation guardrails
- Never run validation or build commands directly (e.g., lint, typecheck, format, test runners). Always use the `validate-changes` skill or the `validator` agent so that validation results stay structured and work is not duplicated.
- When you need to check specific categories after a fix, invoke the validator with the relevant step names (e.g., "types lint") instead of running the full pipeline.
```

If `.claude/CLAUDE.md` does not exist, create it with this content.

### 3e. Confirm

After writing all files, confirm to the user what was created and list:
- The validation steps included in the script
- The files created or updated
- Remind the user that `.claude/scripts/validate.sh` can also be run directly from the terminal or CI

## Important
- Ask questions ONE AT A TIME. Never batch multiple questions.
- Always use `AskUserQuestion` for user input. Do not assume answers.
- If the project has no detectable tooling at all, tell the user and ask them to manually provide commands from the start.
